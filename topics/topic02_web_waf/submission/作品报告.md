## 摘要

随着 Web 应用攻击手段不断演进，攻击者通过 URL 多重编码、Unicode 归一化、SQL AST 拆分、注释注入等混淆手法绕过传统 WAF，对网络安全构成严峻挑战。赛题要求构建能够检测 8 类 Web 攻击（SQLi、XSS、CMD、PathTraversal、FileInclusion、XXE、PromptInjection 等）并在高强度混淆载荷上保持高检出、低误报、可解释、低延迟的防御系统。

本作品 **IGA-Guard 3.0** 提出「多层解混淆 → 四模态条件融合检测 → WebSpotter 可解释定位 → Tip-Adapter 持续学习缓存 → 对抗演化闭环」的完整工程方案。系统在全量 19,411 条诚实评测集上实现混淆子集 Recall **99.95%**（赛题目标 >99.5%）、Normal FPR **1.67%**（P0 护栏后达标 <3%）、混淆 Precision **100%**、单请求延迟 P50 **2.92 ms**、恶意定位 IoU 提升 **+37.9%**，同集 CRS 离线基线混淆 Recall 仅 **54%**。

## 第一章 引言与赛题分析

### 1.1 研究背景

Web 应用防火墙（WAF）是抵御应用层攻击的第一道防线。然而，攻击者日益采用混淆逃逸技术——如双重 URL 编码、空字节注入、SQL CONCAT 拆分、JavaScript 字符串拼接、Unicode 同形字等——使基于静态规则的 WAF 漏检率显著上升。学术界与工业界的研究表明，ModSecurity CRS 在对抗学习混淆下 TPR 可降至 50% 以下，纯机器学习模型在混淆子集上 Recall 普遍下降 20~40 个百分点。

2026 年人工智能安全竞赛题目 2 聚焦「Web 攻击载荷混淆逃逸检测」，要求参赛队伍设计集净化、特征提取、模型检测、对抗模拟于一体的完整系统，并在混淆测试集上达到高检出率与可接受的误报率。

### 1.2 问题陈述

| 挑战 | 描述 |
|------|------|
| 混淆多样性 | 62+ 种混淆手法，组合爆炸 |
| 误报控制 | 正常业务流量（登录表单、URL 参数）易误判 |
| 实时性 | 单请求检测需 ≤10 ms 级延迟 |
| 可解释性 | 安全运维需定位恶意片段，非黑盒判定 |
| 持续演化 | 静态模型无法防御未见变种 |

### 1.3 设计目标

| 维度 | 目标值 | 实测 |
|------|--------|------|
| 混淆子集 Recall | > 99.5% | **99.95%** ✅ |
| 混淆 Precision | 尽可能高 | **100%** ✅ |
| Normal FPR | < 3% | **1.67%** ✅ |
| 延迟 P50 | < 5 ms | **2.92 ms** ✅ |
| 定位 IoU 提升 | ≥ +22% | **+37.9%** ✅ |
| 攻击类型 | 8 类 | SQLi/XSS/CMD/PathTraversal/FileInclusion/XXE/PromptInjection/Normal |

### 2.1 架构概览

```
                    ┌─────────────────────────────────────────────────────────┐
                    │                    IgaGuardEngine                        │
  HTTP Request      │                                                          │
       │            │  collector/     normalizer/      detector/              │
       ▼            │  ┌──────────┐   ┌──────────┐   ┌──────────────┐        │
  parse_http ──────►│  │http_parser│──►│ decoder  │──►│ dual_track   │        │
                    │  │timeseries │   │ast_restore│   │ ├ fusion RF  │        │
                    │  └──────────┘   │(6轮解码)  │   │ ├ semantic   │        │
                    │                 └──────────┘   │ ├ multimodal │        │
                    │                               │ ├ dlinear    │        │
                    │                               │ └ cache      │        │
                    │  explainer/        rules/      └──────┬───────┘        │
                    │  ┌──────────┐     ┌──────────┐         ▼               │
                    │  │webspotter│     │virtual   │   DetectionResult       │
                    │  │nl_explain│     │_patch    │   + highlight_html      │
                    │  └──────────┘     └──────────┘                          │
                    └─────────────────────────────────────────────────────────┘
```

### 2.2 技术栈

| 层级 | 技术选型 |
|------|----------|
| 后端 API | Flask 3.x + flask-cors |
| 检测引擎 | Python 3.10+ · scikit-learn · XGBoost |
| 语义轨 | TinyBERT-6L（本地微调） |
| 时序轨 | DLinear 时序分解异常检测 |
| 前端 | HTML5 + ECharts 六页大屏 |
| 配置 | YAML（`configs/default.yaml`） |

### 2.3 数据流

1. **采集解析**：`http_parser.py` 拆分 URL query、Body、Header、Cookie 为独立 payload 字段
2. **多层解混淆**：最多 6 轮 URL/HTML/Unicode/Hex 解码 + SQL/XSS AST 还原
3. **特征提取**：结构特征（熵、特殊字符密度）+ RL-GWO 筛选 Top-15 维
4. **四模态融合**：RF+规则 / TinyBERT / 协议+字节图 / DLinear 条件加权
5. **缓存修正**：552 条 Tip-Adapter KV 缓存相似度查库
6. **FP 护栏**：抑制缓存/规则翻判导致的误报
7. **可解释输出**：WebSpotter 字符级 span 高亮 + 自然语言解释

### 2.4 四模态条件融合

| 模态 | 模块 | 非混淆权重 | 混淆权重 | 作用 |
|------|------|-----------|----------|------|
| 统计轨 | RF + 结构规则 | 38% | 42% | 基线检出 + 规则兜底 |
| 语义轨 | TinyBERT | 28% | 32% | 解混淆后语义分类 |
| 多模态轨 | 协议 + 字节图 | 14% | **4%** | 混淆时降权压 FP |
| 时序轨 | DLinear | 12% | 12% | 同 IP 请求序列异常 |
| 持续学习 | KV 缓存 | fusion 0.28 | — | 漏检扩库记忆 |

**核心设计**：混淆样本压低多模态权重，避免视觉轨对编码流量的过拟合误报；良性流量加强多模态压 FP。`is_obfuscated()`（评测口径）与 `has_strong_obfuscation()`（检测器专用）双层判定，避免普通 URL 编码误触发 Boost。

## 第三章 核心模块实现

### 3.1 智能解混淆模块（`normalizer/`）

- **decoder.py**：6 轮迭代解码链（URL 单/双编码、HTML 实体、Unicode、Hex、Base64）
- **ast_restore.py**：SQL `CONCAT`/`CHAR` 还原、XSS 事件处理器剥离
- 记录 `decode_chain` 深度，供混淆 Boost 判定

### 3.2 四模态检测引擎（`detector/dual_track.py`）

- **FusionDetector**：128 棵 Random Forest + `structural_attack_rules()` 结构规则
- **SemanticBranch**：TinyBERT 本地推理，无 GPU 时关键词密度回退
- **MultimodalBranch**：HPP、multipart 协议特征 + 32×64 字节图视觉轨
- **DLinearBranch**：按 `src_ip` 维护 T=16 步时序矩阵，输出异常分
- **混淆 Boost**：强混淆 + 深解码 + base 攻击峰值达标时抬升攻击分
- **FP 护栏 Tier A/B/C**：RF 判 Normal 却被缓存翻判时要求更高置信

### 3.3 混淆逃逸规则引擎（`obfuscation_signals.py`）

- **62 种混淆手法标记**：`OBFUSCATED_MARKERS` 元组
- **obfuscated_evasion_rescue**：针对 null_byte、echo、concat、hex32、url_encode 等高频漏检模式的兜底规则
- **looks_like_benign_csic_form**：识别 CSIC 正常登录表单，压制 SQLi 误报

### 3.4 Tip-Adapter 持续学习缓存（`evolution/continual_cache.py`）

- 552 条双 Key（文本 + 视觉）KV 缓存
- 漏检样本通过 `expand-cache` 扩库，无需全量重训
- 良性流量 `fusion_weight` 降至 0.12，强制命中需 hit≥0.92

### 3.5 可解释性模块（`explainer/webspotter.py`）

- 字段贡献度条形图 + 字符级恶意 span
- `build_highlight_html()` 生成前端红色 `<mark>` 高亮
- `nl_explanation.py` 自然语言模板解释

### 3.6 对抗演化闭环

```
载荷种子 → obfuscate (62种手法) → detect → 漏检? → feedback → expand-cache → evolve → Online RL 调阈值
```

- `adversarial/mutator.py` + `ast_mutator.py` + `llm_agent.py`
- `scripts/run_adversarial.py` 多轮 E3 对抗实验

### 3.7 Web 服务与 API（`backend/app.py`）

| 端点 | 方法 | 功能 |
|------|------|------|
| `/api/detect` | POST | 单条 HTTP 请求检测 |
| `/api/obfuscate` | POST | 混淆变种生成 |
| `/api/feedback` | POST | 漏检/误报反馈 |
| `/api/evolve` | POST | 在线演化 |
| `/api/alerts` | GET | 最近告警 |
| `/api/metrics/overall` | GET | 离线评测指标 |

前端六页：`hub` · `p1_monitor` · `p2_detail` · `p3_localization` · `p4_risk` · `p5_evolution` · `p6_rules`

## 第六章 创新点

### 创新点一：四模态条件融合 + 强/弱混淆分层

RF+规则、TinyBERT、协议+字节图、DLinear 四路 Late Fusion，按混淆/非混淆动态分权。优于纯 BERT WAF（无协议/时序轨）与 ModSecurity CRS（无 ML 门控）。

### 创新点二：Tip-Adapter 持续学习缓存

冻结编码器 + 动态漏检扩库（552 条），无需全量重训即可吸收新变种。优于静态 RF/XGBoost 与 CNN-LSTM 全量重训成本。

### 创新点三：混淆逃逸规则引擎 + 分层 FP 护栏

`obfuscated_evasion_rescue` 五类高频漏检兜底 + CSIC 良性表单识别 + Tier A/B/C FP 护栏。将混淆漏检从数百条压至 5 条，同时控制误报翻判。

### 创新点四：WebSpotter 可解释定位 + 六页演示大屏

字段贡献 + 字符 span 高亮 + NL 解释，IoU 提升 +37.9%。优于 CNN-LSTM/BERT 黑盒（无可解释定位）。

### 创新点五：62 种混淆手法库 + 对抗演化闭环

`obfuscation_techniques.py` 可编程合成 + LLM/AST 对抗 + 漏检反馈 → 缓存扩库 → Online RL 阈值调整。满足赛题混淆载荷生成器要求。

## 第八章 总结与展望

### 8.1 工作总结

IGA-Guard 3.0 完成了从数据采集、多层解混淆、四模态融合检测、可解释定位到对抗演化闭环的完整工程实现。在赛题核心指标上，混淆子集 Recall 达到 **99.95%**，Normal FPR **1.67%**，延迟 P50 **2.92 ms**，定位 IoU 提升 **+37.9%**。

### 8.2 不足与改进

1. P2 LLM 红队全量实验待 Ollama 接入（规则回退模式 pooled recall 76.7%）
2. P6 INT8/ONNX 语义轨加速以降低 P99 长尾（当前 27ms）
3. 手法 DSL 已落地，可继续扩展至 LLM 自动生成 transform
2. P99 延迟长尾偏高（模型预热与量化）
3. 剩余 5 条极端混淆漏检（concat/hex 边界）

### 8.3 应用前景

- 企业 WAF 旁路检测与 SOC 告警解释
- 红蓝对抗演练与混淆手法研究
- 虚拟补丁快速响应（Log4Shell 等）
